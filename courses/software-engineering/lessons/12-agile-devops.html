Agile Methodologies & DevOps Practices

Master Agile principles, Scrum framework, Kanban methodology, and DevOps practices including CI/CD, containerization, and cloud platforms.

1. Agile Manifesto and Principles

Agile is a mindset and set of values for software development that emphasizes flexibility, collaboration, and customer satisfaction.

The Agile Manifesto (2001):

We value:

  • Individuals and interactions over processes and tools
  • Working software over comprehensive documentation
  • Customer collaboration over contract negotiation
  • Responding to change over following a plan

That is, while there is value in the items on the right, we value the items on the left more.

12 Agile Principles:
  1. Customer satisfaction through early and continuous delivery
  2. Welcome changing requirements, even late in development
  3. Deliver working software frequently (weeks rather than months)
  4. Business people and developers work together daily
  5. Build projects around motivated individuals
  6. Face-to-face conversation is most effective
  7. Working software is primary measure of progress
  8. Sustainable development - maintain constant pace
  9. Continuous attention to technical excellence
  10. Simplicity - maximize work not done
  11. Self-organizing teams produce best results
  12. Regular reflection and adjustment

2. Scrum Framework

Scrum is an Agile framework for managing complex projects with iterative development cycles called sprints.

2.1 Scrum Roles

Three Core Roles:
  • Product Owner: Defines features, prioritizes backlog, accepts/rejects work
  • Scrum Master: Facilitates process, removes impediments, coaches team
  • Development Team: Self-organizing, cross-functional, 3-9 people

2.2 Scrum Artifacts

Scrum Artifacts 
1. PRODUCT BACKLOG
   - Prioritized list of features
   - User stories with acceptance criteria
   - Maintained by Product Owner

   Example User Story:
   "As a user, I want to reset my password via email
    so that I can regain access if I forget it"
   
   Acceptance Criteria:
   - User receives reset email within 5 minutes
   - Reset link expires after 24 hours
   - Old password no longer works after reset

2. SPRINT BACKLOG
   - Selected items from product backlog
   - Tasks for current sprint
   - Owned by development team

   Sprint Goal: "Implement user authentication features"
   
   Tasks:
   - [ ] Design login UI
   - [ ] Implement password validation
   - [ ] Add forgot password functionality
   - [ ] Write unit tests
   - [ ] Update documentation

3. INCREMENT
   - Potentially shippable product
   - Sum of all completed backlog items
   - Must meet Definition of Done

   Definition of Done:
   âœ“ Code reviewed
   âœ“ Unit tests pass (>80% coverage)
   âœ“ Integration tests pass
   âœ“ Documentation updated
   âœ“ Deployed to staging
   âœ“ Accepted by Product Owner

2.3 Scrum Ceremonies

Sprint Events 
1. SPRINT PLANNING (2-4 hours for 2-week sprint)
   - What will be delivered?
   - How will work be achieved?
   
   Input: Product backlog, team velocity
   Output: Sprint goal, sprint backlog
   
   Activities:
   - Product Owner presents top priorities
   - Team discusses and estimates stories
   - Team commits to sprint goal

2. DAILY STANDUP (15 minutes)
   - What did I do yesterday?
   - What will I do today?
   - Any blockers?
   
   Rules:
   - Same time, same place
   - Stand up (keeps it short)
   - Focus on commitment, not status report

3. SPRINT REVIEW (1-2 hours)
   - Demo completed work
   - Stakeholder feedback
   - Update product backlog
   
   Attendees: Team, Product Owner, stakeholders
   Format: Live demo, not PowerPoint

4. SPRINT RETROSPECTIVE (1-1.5 hours)
   - What went well?
   - What could be improved?
   - Action items for next sprint
   
   Techniques:
   - Start, Stop, Continue
   - Glad, Sad, Mad
   - 4 L's (Liked, Learned, Lacked, Longed for)

2.4 Sprint Cycle

Typical 2-Week Sprint 
Week 1:
Monday:
  9:00 AM  - Sprint Planning (4 hours)
  1:00 PM  - Team starts development

Tuesday-Friday:
  9:00 AM  - Daily Standup (15 min)
  Rest of day - Development

Week 2:
Monday-Thursday:
  9:00 AM  - Daily Standup (15 min)
  Rest of day - Development

Friday:
  9:00 AM  - Daily Standup (15 min)
  10:00 AM - Sprint Review (2 hours)
  1:00 PM  - Sprint Retrospective (1.5 hours)
  3:00 PM  - Backlog refinement (optional)

Sprint Velocity Example:
Sprint 1: 23 story points completed
Sprint 2: 27 story points completed
Sprint 3: 25 story points completed
Average: 25 points/sprint

Next sprint planning: Commit to ~25 points

4. DevOps Culture and Practices

DevOps is a culture and set of practices that combines software development (Dev) and IT operations (Ops) to shorten development lifecycle and deliver high-quality software continuously.

DevOps Principles:
  • Collaboration: Break down silos between Dev and Ops
  • Automation: Automate repetitive tasks
  • Continuous Integration: Merge code frequently
  • Continuous Delivery: Release often and reliably
  • Monitoring: Observe and measure everything
  • Feedback: Fast feedback loops

Continuous Integration (CI):
1. Developer commits code
2. CI server detects change
3. Code is built
4. Tests run automatically
5. Feedback sent to developer

Continuous Delivery (CD):
6. Deploy to staging automatically
7. Manual approval for production

Continuous Deployment:
6-7. Deploy to production automatically
GitHub Actions CI/CD Example 
# .github/workflows/ci-cd.yml
name: CI/CD Pipeline

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    
    steps:
    - name: Checkout code
      uses: actions/checkout@v3

    - name: Setup Node.js
      uses: actions/setup-node@v3
      with:
        node-version: '18'
        cache: 'npm'

    - name: Install dependencies
      run: npm ci

    - name: Run linter
      run: npm run lint

    - name: Run tests
      run: npm test -- --coverage

    - name: Build application
      run: npm run build

    - name: Upload coverage
      uses: codecov/codecov-action@v3

  deploy-staging:
    needs: build-and-test
    if: github.ref == 'refs/heads/develop'
    runs-on: ubuntu-latest
    
    steps:
    - name: Deploy to staging
      run: |
        echo "Deploying to staging..."
        # Deploy commands here

  deploy-production:
    needs: build-and-test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment: production
    
    steps:
    - name: Deploy to production
      run: |
        echo "Deploying to production..."
        # Deploy commands here

    - name: Notify team
      run: |
        echo "Sending deployment notification..."

5. Containerization with Docker

Containers package applications with their dependencies, ensuring consistency across environments.

Dockerfile Example 
# Dockerfile
# Multi-stage build for Node.js application

# Stage 1: Build
FROM node:18-alpine AS builder

WORKDIR /app

# Copy package files
COPY package*.json ./

# Install dependencies
RUN npm ci --only=production

# Copy source code
COPY . .

# Build application
RUN npm run build

# Stage 2: Production
FROM node:18-alpine

WORKDIR /app

# Copy built app from builder
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY package*.json ./

# Create non-root user
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nodejs -u 1001

USER nodejs

# Expose port
EXPOSE 3000

# Health check
HEALTHCHECK --interval=30s --timeout=3s \
  CMD node healthcheck.js

# Start application
CMD ["node", "dist/server.js"]
Docker Commands 
# Build image
docker build -t myapp:latest .

# Run container

# View running containers

6. Kubernetes Basics

Kubernetes (K8s) is a container orchestration platform for automating deployment, scaling, and management of containerized applications.

Kubernetes Resources 
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  labels:
    app: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: myapp:latest
        ports:
        - containerPort: 3000
        env:
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: url
        resources:
          requests:
            memory: "128Mi"
            cpu: "100m"
          limits:
            memory: "256Mi"
            cpu: "200m"
        livenessProbe:
          httpGet:
            path: /health
            port: 3000
          initialDelaySeconds: 30
          periodSeconds: 10

---
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp-service
spec:
  selector:
    app: myapp
  ports:
  - protocol: TCP
    port: 80
    targetPort: 3000
  type: LoadBalancer

---
# ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt
spec:
  rules:
  - host: myapp.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: myapp-service
            port:
              number: 80
  tls:
  - hosts:
    - myapp.com
    secretName: myapp-tls

Problem 1: Plan a Sprint

Scenario: You're starting a 2-week sprint with a team of 5 developers. Last 3 sprints completed 25, 28, and 27 story points.

  • Define sprint goal

Product Backlog:

  • User authentication (8 points)
  • Password reset (5 points)
  • User profile page (8 points)
  • Search functionality (13 points)
  • Email notifications (5 points)
  • Admin dashboard (13 points)
Show Solution

Problem 2: Design CI/CD Pipeline

Task: Design a CI/CD pipeline for a Node.js application that deploys to staging and production.

Show Solution 
Pipeline Stages:

1. CODE COMMIT
   Trigger: Push to any branch

2. BUILD & TEST (All branches)
   - Install dependencies
   - Run linter
   - Run unit tests
   - Run integration tests
   - Generate coverage report
   - Build application

3. SECURITY SCAN
   - Dependency vulnerability scan
   - Static code analysis
   - Secret detection

4. DEPLOY TO STAGING (develop branch only)
   - Build Docker image
   - Push to registry
   - Deploy to staging environment
   - Run smoke tests

5. MANUAL APPROVAL (main branch only)
   - Product Owner approval
   - QA sign-off

6. DEPLOY TO PRODUCTION (main branch after approval)
   - Blue-green deployment
   - Health checks
   - Rollback capability
   - Notify team

7. POST-DEPLOYMENT
   - Monitor metrics
   - Check error rates
   - Alert on anomalies

- Container Registry: Docker Hub / ECR
- Orchestration: Kubernetes
- Monitoring: Prometheus + Grafana

Key Takeaways

  • Agile: Flexible, iterative development focused on customer collaboration and responding to change
  • Scrum: Fixed sprints, defined roles (PO, SM, Team), ceremonies (planning, standup, review, retro)
  • Kanban: Continuous flow, WIP limits, visualize workflow, flexible and adaptive
  • DevOps: Culture of collaboration between Dev and Ops, automation, continuous delivery
  • CI/CD: Automate build, test, and deployment - fast feedback and reliable releases
  • Docker: Containerize applications for consistency across environments
  • Kubernetes: Orchestrate containers at scale with automated deployment and scaling
  • Monitoring: Observe system behavior, measure performance, detect issues early
  • Cloud Platforms: AWS, Azure, GCP provide scalable infrastructure and services
  • DevOps is a culture, not just tools - collaboration and communication are key
