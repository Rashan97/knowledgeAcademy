Lesson 4: Creational Design Patterns

üìñ 60 min read üìä Intermediate

Introduction to Creational Patterns

Creational design patterns deal with object creation mechanisms, providing flexibility in deciding which objects to create for a given use case. They abstract the instantiation process, making the system independent of how its objects are created, composed, and represented.

üìå Why Creational Patterns Matter

Direct object creation using constructors can lead to tight coupling and inflexibility. Creational patterns provide controlled, flexible ways to create objects that promote loose coupling and code reusability.

The 5 Creational Patterns

  • Singleton - Ensure a class has only one instance
  • Factory Method - Define an interface for creating objects
  • Abstract Factory - Create families of related objects
  • Builder - Construct complex objects step by step
  • Prototype - Create objects by cloning existing ones

1. Singleton Pattern

The Singleton pattern ensures a class has only one instance and provides a global point of access to it. This is useful for resources that should only exist once, like configuration managers, connection pools, or logging systems.

Problem It Solves

  • Need exactly one instance of a class
  • Global access to that instance
  • Lazy initialization (create only when needed)
  • Thread-safe access in concurrent environments

Implementation in Python

class DatabaseConnection: # Class variable to hold the single instance _instance = None _lock = threading.Lock() def __new__(cls): # Thread-safe singleton implementation if cls._instance is None: with cls._lock: # Double-check locking if cls._instance is None: cls._instance = super().__new__(cls) cls._instance._initialized = False return cls._instance def __init__(self): # Initialize only once if not self._initialized: self.connection = self._create_connection() self._initialized = True def _create_connection(self): print("Creating database connection...") return "DB Connection Object" # Usage db1 = DatabaseConnection() db2 = DatabaseConnection() print(db1 is db2) # True - same instance!

Implementation in JavaScript

class Logger { constructor() { if (Logger.instance) { return Logger.instance; } this.logs = []; Logger.instance = this; } log(message) { this.logs.push({ message, timestamp: new Date() }); console.log(`[LOG] ${message}`); } printLogs() { this.logs.forEach(log => { console.log(`${log.timestamp}: ${log.message}`); }); } } // Usage const logger1 = new Logger(); const logger2 = new Logger(); console.log(logger1 === logger2); // true

‚ö†Ô∏è Singleton Pitfalls

  • Can make testing difficult (global state)
  • May violate Single Responsibility Principle
  • Can hide dependencies
  • Thread-safety issues if not implemented carefully

When to Use Singleton

  • Configuration managers
  • Logging systems
  • Database connection pools
  • Thread pools
  • Cache management

2. Factory Method Pattern

The Factory Method defines an interface for creating objects but lets subclasses decide which class to instantiate. It delegates the instantiation logic to subclasses.

Problem It Solves

  • Create objects without specifying their exact classes
  • Delegate instantiation to subclasses
  • Promote loose coupling
  • Follow Open/Closed Principle (open for extension, closed for modification)

Implementation Example

from abc import ABC, abstractmethod # Product Interface class Vehicle(ABC): @abstractmethod def deliver(self): pass # Concrete Products class Truck(Vehicle): def deliver(self): return "Delivering by land in a truck" class Ship(Vehicle): def deliver(self): return "Delivering by sea in a ship" class Plane(Vehicle): def deliver(self): return "Delivering by air in a plane" # Creator (Factory) class Logistics(ABC): @abstractmethod def create_transport(self) -> Vehicle: pass def plan_delivery(self): # Factory method is called here vehicle = self.create_transport() return vehicle.deliver() # Concrete Creators class RoadLogistics(Logistics): def create_transport(self): return Truck() class SeaLogistics(Logistics): def create_transport(self): return Ship() class AirLogistics(Logistics): def create_transport(self): return Plane() # Usage def client_code(logistics: Logistics): print(logistics.plan_delivery()) client_code(RoadLogistics()) # Delivering by land in a truck client_code(SeaLogistics()) # Delivering by sea in a ship client_code(AirLogistics()) # Delivering by air in a plane

üí° Real-World Example

UI frameworks use Factory Method extensively. For example, a GUI framework might have different factories for Windows, macOS, and Linux, each creating platform-specific buttons, checkboxes, and windows while maintaining the same interface.

3. Abstract Factory Pattern

Abstract Factory provides an interface for creating families of related or dependent objects without specifying their concrete classes. It's essentially a factory of factories.

Implementation Example - UI Components

# Abstract Products class Button(ABC): @abstractmethod def render(self): pass class Checkbox(ABC): @abstractmethod def render(self): pass # Windows Concrete Products class WindowsButton(Button): def render(self): return "Rendering Windows-style button" class WindowsCheckbox(Checkbox): def render(self): return "Rendering Windows-style checkbox" # macOS Concrete Products class MacButton(Button): def render(self): return "Rendering macOS-style button" class MacCheckbox(Checkbox): def render(self): return "Rendering macOS-style checkbox" # Abstract Factory class GUIFactory(ABC): @abstractmethod def create_button(self) -> Button: pass @abstractmethod def create_checkbox(self) -> Checkbox: pass # Concrete Factories class WindowsFactory(GUIFactory): def create_button(self): return WindowsButton() def create_checkbox(self): return WindowsCheckbox() class MacFactory(GUIFactory): def create_button(self): return MacButton() def create_checkbox(self): return MacCheckbox() # Client Code class Application: def __init__(self, factory: GUIFactory): self.button = factory.create_button() self.checkbox = factory.create_checkbox() def render(self): print(self.button.render()) print(self.checkbox.render()) # Usage import platform if platform.system() == "Windows": factory = WindowsFactory() elif platform.system() == "Darwin": factory = MacFactory() app = Application(factory) app.render()

Key Differences: Factory Method vs Abstract Factory

Aspect Factory Method Abstract Factory
Purpose Create one product Create families of products
Complexity Simple More complex
Inheritance Uses inheritance Uses composition
Example Create different vehicles Create entire UI themes

4. Builder Pattern

Builder pattern separates the construction of a complex object from its representation, allowing the same construction process to create different representations.

Problem It Solves

  • Too many constructor parameters (telescoping constructor anti-pattern)
  • Need to create objects with many optional parameters
  • Step-by-step object construction
  • Immutable objects with many fields

Implementation Example

class Pizza: def __init__(self): self.size = None self.cheese = False self.pepperoni = False self.mushrooms = False self.olives = False self.bacon = False def __str__(self): toppings = [] if self.cheese: toppings.append("cheese") if self.pepperoni: toppings.append("pepperoni") if self.mushrooms: toppings.append("mushrooms") if self.olives: toppings.append("olives") if self.bacon: toppings.append("bacon") return f"{self.size} pizza with {', '.join(toppings)}" class PizzaBuilder: def __init__(self): self.pizza = Pizza() def set_size(self, size): self.pizza.size = size return self # Return self for method chaining def add_cheese(self): self.pizza.cheese = True return self def add_pepperoni(self): self.pizza.pepperoni = True return self def add_mushrooms(self): self.pizza.mushrooms = True return self def add_olives(self): self.pizza.olives = True return self def add_bacon(self): self.pizza.bacon = True return self def build(self): return self.pizza # Usage - Fluent Interface pizza = (PizzaBuilder() .set_size("Large") .add_cheese() .add_pepperoni() .add_mushrooms() .build()) print(pizza) # Large pizza with cheese, pepperoni, mushrooms

JavaScript Implementation with Director

class Computer { constructor() { this.cpu = null; this.ram = null; this.storage = null; this.gpu = null; } } class ComputerBuilder { constructor() { this.computer = new Computer(); } setCPU(cpu) { this.computer.cpu = cpu; return this; } setRAM(ram) { this.computer.ram = ram; return this; } setStorage(storage) { this.computer.storage = storage; return this; } setGPU(gpu) { this.computer.gpu = gpu; return this; } build() { return this.computer; } } // Director - knows how to build common configurations class ComputerDirector { static buildGamingPC() { return new ComputerBuilder() .setCPU("Intel i9-13900K") .setRAM("32GB DDR5") .setStorage("2TB NVMe SSD") .setGPU("RTX 4090") .build(); } static buildOfficePC() { return new ComputerBuilder() .setCPU("Intel i5-13400") .setRAM("16GB DDR4") .setStorage("512GB SSD") .build(); // No GPU needed } } // Usage const gamingPC = ComputerDirector.buildGamingPC(); const officePC = ComputerDirector.buildOfficePC();

üí° When to Use Builder

Use Builder when you have a class with more than 4-5 constructor parameters, especially if many are optional. Common in configuration objects, HTTP request builders, and query builders (SQL, NoSQL).

5. Prototype Pattern

Prototype pattern creates new objects by cloning existing ones rather than creating from scratch. Useful when object creation is expensive or complex.

Implementation Example

import copy class Document: def __init__(self, title, content, formatting): self.title = title self.content = content self.formatting = formatting # Complex object print(f"Creating document: {title}") def clone(self): # Deep copy to avoid shared references return copy.deepcopy(self) def __str__(self): return f"Document: {self.title}" # Create expensive template template = Document( title="Template", content="Lorem ipsum...", formatting={ "font": "Arial", "size": 12, "colors": {"primary": "#000", "secondary": "#666"} } ) # Clone instead of creating from scratch doc1 = template.clone() doc1.title = "Report Q1" doc2 = template.clone() doc2.title = "Report Q2" # No expensive creation logs for clones!

Summary: Creational Patterns Comparison

Pattern When to Use Example
Singleton Need exactly one instance Configuration, Logger, Cache
Factory Method Subclasses decide what to create Transport logistics, Document creators
Abstract Factory Create families of related objects UI toolkits, Database drivers
Builder Complex object construction Pizza, Computer, HTTP Request
Prototype Cloning is cheaper than creating Document templates, Game objects

üéØ Key Takeaways

  • Creational patterns provide flexibility in object creation
  • Choose based on your specific needs, not arbitrarily
  • They promote loose coupling and code reusability
  • Can be combined (e.g., Singleton Factory)
  • Understand trade-offs - added complexity vs. flexibility
