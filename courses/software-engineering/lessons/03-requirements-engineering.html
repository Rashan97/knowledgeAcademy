Requirements Engineering

Learn how to elicit, analyze, document, and validate software requirements - the foundation of successful software projects.

1. Introduction to Requirements Engineering

What are Requirements?

Requirements define what a system should do (functionality) and under what constraints it must operate (quality attributes). They form a contract between stakeholders and developers.

Requirement Types:
  • Functional Requirements: Describe what the system should do (features, behaviors, inputs/outputs)
  • Non-Functional Requirements (NFRs): Describe how the system should perform (performance, security, usability)
  • Domain Requirements: Specific to the application domain (e.g., regulatory compliance)

Requirements Engineering Process

The RE process consists of four main activities:

  1. Elicitation: Discovering requirements from stakeholders
  2. Analysis: Refining and organizing requirements
  3. Specification: Documenting requirements formally
  4. Validation: Ensuring requirements meet stakeholder needs

2. Functional Requirements

Characteristics of Good Functional Requirements

Functional requirements should be:

  • Complete: All necessary functionality is specified
  • Consistent: No contradictions between requirements
  • Unambiguous: Only one interpretation possible
  • Verifiable: Can be tested/verified
  • Traceable: Linked to business needs and design elements
Example - Banking System: 
FR-001: The system shall allow users to transfer funds between accounts
FR-002: The system shall verify sufficient balance before processing transfers
FR-003: The system shall generate a transaction confirmation number
FR-004: The system shall send email notification upon successful transfer
FR-005: The system shall log all transaction attempts with timestamp

Writing Functional Requirements

Use the template: "The system shall [action] [object] [condition]"

Good vs. Bad Requirements: 
‚ùå Bad: "The system should be fast"
‚úÖ Good: "The system shall process login requests within 2 seconds"

‚ùå Bad: "Users can search for products"
‚úÖ Good: "The system shall allow authenticated users to search products by name, category, or SKU"

‚ùå Bad: "Handle errors properly"
‚úÖ Good: "The system shall display a user-friendly error message when database connection fails"

3. Non-Functional Requirements (NFRs)

NFR Categories

Performance Requirements:

  • Response Time: Max time for system response (e.g., < 3 seconds for page load)
  • Throughput: Transactions per second (e.g., 10,000 TPS)
  • Resource Usage: CPU, memory, bandwidth limits

Scalability Requirements:

  • Concurrent Users: Max simultaneous users (e.g., 100,000 concurrent users)
  • Data Volume: Storage capacity (e.g., support 1TB database)
  • Load Growth: Future expansion (e.g., 50% annual user growth)

Security Requirements:

  • Authentication: How users are verified (e.g., OAuth 2.0, MFA)
  • Authorization: Access control (e.g., role-based access control)
  • Data Encryption: At rest and in transit (e.g., AES-256, TLS 1.3)
  • Audit Logging: Track security-relevant events

Usability Requirements:

  • Learnability: Time for new users to become productive
  • Accessibility: WCAG 2.1 Level AA compliance
  • Error Prevention: Input validation, confirmation dialogs

Reliability Requirements:

  • Availability: Uptime percentage (e.g., 99.9% = 8.76 hours downtime/year)
  • MTBF (Mean Time Between Failures): Average operational time
  • MTTR (Mean Time To Repair): Average recovery time
  • Fault Tolerance: Continue operation despite component failures
Availability Calculation: $$\text{Availability} = \frac{\text{MTBF}}{\text{MTBF} + \text{MTTR}} \times 100\%$$

Example: If MTBF = 720 hours and MTTR = 0.5 hours:

$$\text{Availability} = \frac{720}{720 + 0.5} \times 100\% = 99.93\%$$
NFR Examples: 
NFR-001: The system shall support 50,000 concurrent users
NFR-002: The system shall achieve 99.95% uptime (4.38 hours downtime/year)
NFR-003: All API responses shall complete within 500ms at 95th percentile
NFR-004: The system shall encrypt all data using AES-256 encryption
NFR-005: The system shall comply with GDPR and CCPA regulations
NFR-006: The web interface shall be accessible per WCAG 2.1 Level AA
NFR-007: The system shall scale horizontally to handle 100% traffic increase

4. Use Cases

Use Case Anatomy

A use case describes interactions between actors (users/systems) and the system to achieve a goal.

Use Case Structure:
  • Title: Goal in brief (e.g., "Process Customer Order")
  • Actors: Primary (initiates) and Secondary (supports)
  • Preconditions: Must be true before execution
  • Main Success Scenario: Step-by-step happy path
  • Extensions: Alternative paths and error handling
  • Postconditions: System state after completion
Use Case Example: ATM Cash Withdrawal 
Use Case: Withdraw Cash
Primary Actor: Bank Customer
Preconditions: Customer has valid ATM card and account with sufficient balance
Postconditions: Cash dispensed, account debited, transaction recorded

Main Success Scenario:
1. Customer inserts ATM card
2. System prompts for PIN
3. Customer enters PIN
4. System validates PIN
5. System displays transaction menu
6. Customer selects "Withdraw Cash"
7. System prompts for amount
8. Customer enters amount ($100)
9. System verifies sufficient balance
10. System dispenses cash
11. System prints receipt
12. System ejects card
13. Customer removes card and cash

Extensions:
4a. Invalid PIN:
    4a1. System displays error message
    4a2. System prompts for PIN again
    4a3. After 3 failed attempts, system retains card
    
9a. Insufficient balance:
    9a1. System displays "Insufficient funds" message
    9a2. System returns to step 7
    
10a. Cash dispenser empty:
    10a1. System displays "Service unavailable" message
    10a2. System logs maintenance alert
    10a3. Use case ends

Use Case Diagrams

Visual representation using UML notation showing actors, use cases, and relationships:

  • Include: Required sub-functionality (e.g., "Login" included in multiple use cases)
  • Extend: Optional behavior (e.g., "Apply Discount" extends "Process Order")
  • Generalization: Inheritance between actors or use cases

5. User Stories (Agile Approach)

User Story Format

A lightweight alternative to use cases, popular in Agile methodologies:

Standard Template:

As a [role], I want [feature], so that [benefit]

User Story Examples: 
As a customer, I want to save items to a wishlist, so that I can purchase them later

As an admin, I want to export user data to CSV, so that I can analyze trends in Excel

As a mobile user, I want to use fingerprint authentication, so that I can log in quickly

As a content creator, I want to schedule posts, so that I can publish during peak hours

INVEST Criteria

Good user stories should be:

  • Independent: Can be developed in any order
  • Negotiable: Details can be discussed
  • Valuable: Delivers value to users
  • Estimable: Team can estimate effort
  • Small: Completable in one sprint
  • Testable: Clear acceptance criteria

Acceptance Criteria

Define "done" for a user story using Given-When-Then format:

Example with Acceptance Criteria: 
User Story:
As a registered user, I want to reset my password, so that I can regain access if I forget it

Acceptance Criteria:
Given I am on the login page
When I click "Forgot Password"
Then I should see a password reset form

Given I enter my registered email
When I submit the reset form
Then I should receive a reset link via email within 5 minutes

Given I click the reset link
When the link is less than 24 hours old
Then I should see a new password creation form

Given I enter and confirm a new password
When the password meets security requirements (8+ chars, uppercase, lowercase, number)
Then my password should be updated and I should be redirected to login

6. Requirements Elicitation Techniques

Interview Techniques

Structured Interviews: Predefined questions, easy to analyze

Unstructured Interviews: Open-ended, exploratory

Tips:

  • Prepare questions in advance
  • Ask open-ended questions ("How do you...?" vs "Do you...?")
  • Listen actively, probe for details
  • Document immediately after

Workshops and Facilitated Sessions

Joint Application Development (JAD): Structured workshops with stakeholders

Brainstorming: Generate many ideas quickly

Focus Groups: Gather diverse perspectives

Observation

Watch users perform tasks in their natural environment:

  • Ethnographic Studies: Immerse in user environment
  • Contextual Inquiry: Observe and interview simultaneously
  • Identify implicit requirements users can't articulate

Prototyping

Build early versions to explore requirements:

  • Throwaway Prototypes: Quick mockups, discarded after feedback
  • Evolutionary Prototypes: Incrementally refined into final system
  • Wireframes/Mockups: UI-focused prototypes

Document Analysis

Review existing documentation:

  • Current system manuals
  • Business process documents
  • Regulatory requirements
  • Competitor analysis

Surveys and Questionnaires

Gather data from large groups:

  • Use for quantitative data (priorities, preferences)
  • Keep short and focused
  • Mix question types (multiple choice, Likert scale, open-ended)

7. Requirements Analysis and Modeling

Requirements Prioritization

MoSCoW Method:

  • Must have: Critical, non-negotiable
  • Should have: Important but not critical
  • Could have: Desirable if time/budget permits
  • Won't have (this time): Deferred to future releases

Kano Model:

  • Basic Needs: Expected features (cause dissatisfaction if missing)
  • Performance Needs: More is better (linear satisfaction)
  • Delight Needs: Unexpected features (high satisfaction)

Value vs. Effort Matrix:

High Value, Low Effort  ‚Üí Quick Wins (do first)
High Value, High Effort ‚Üí Major Projects (plan carefully)
Low Value, Low Effort   ‚Üí Fill-ins (do if time available)
Low Value, High Effort  ‚Üí Time Wasters (avoid)

Data Modeling

Entity-Relationship Diagrams (ERD): Model data structure

  • Entities: Objects/concepts (Customer, Order, Product)
  • Attributes: Properties (Customer: name, email, phone)
  • Relationships: Associations with cardinality (one-to-many, many-to-many)

Behavioral Modeling

State Diagrams: Model object lifecycle

Activity Diagrams: Model workflows and business processes

Sequence Diagrams: Model interactions between components over time

8. Requirements Specification

Software Requirements Specification (SRS)

Formal document containing all requirements. IEEE 830 standard structure:

  1. Introduction: Purpose, scope, definitions, references
  2. Overall Description: Product perspective, functions, user characteristics, constraints
  3. Specific Requirements: Functional, non-functional, interface requirements
  4. Appendices: Glossary, analysis models, supporting information

Writing Clear Requirements

Avoid ambiguity:

‚ùå Ambiguous words: "fast", "user-friendly", "robust", "flexible"
‚úÖ Specific: "within 2 seconds", "5 clicks or less", "99.9% uptime"

‚ùå "The system shall provide adequate security"
‚úÖ "The system shall use TLS 1.3 for all network communications"

‚ùå "The system should handle many users"
‚úÖ "The system shall support 10,000 concurrent users with <1s response time"

Use consistent terminology:

  • Create a glossary
  • Use domain-specific terms correctly
  • Avoid synonyms (pick one term: user/customer/client)

Requirement numbering:

FR-AUTH-001: Login functionality
FR-AUTH-002: Password reset
FR-SRCH-001: Product search
NFR-PERF-001: Response time requirement
NFR-SEC-001: Encryption requirement

9. Requirements Validation

Validation Techniques

Reviews and Inspections:

  • Walkthroughs: Author presents requirements, team asks questions
  • Inspections: Formal process with defined roles (moderator, reader, recorder)
  • Peer Reviews: Colleagues review for errors

Prototyping Validation:

  • Build prototype based on requirements
  • Users interact and provide feedback
  • Identify missing/incorrect requirements early

Test Case Generation:

  • Write test cases from requirements
  • If you can't test it, requirement is incomplete/ambiguous
  • Ensures verifiability

Requirements Traceability Matrix (RTM):

Links requirements to other artifacts (design, code, tests):

Example RTM:
Req ID Description Design Element Code Module Test Case
FR-001 User login AuthController auth.py TC-AUTH-01
NFR-001 Response time < 2s Caching layer cache.py TC-PERF-01

Validation Checklist

  • ‚úì Are all requirements consistent (no contradictions)?
  • ‚úì Are requirements complete (nothing missing)?
  • ‚úì Are requirements realistic and achievable?
  • ‚úì Are requirements verifiable/testable?
  • ‚úì Are requirements traceable to business needs?
  • ‚úì Do stakeholders agree with requirements?

10. Requirements Management

Change Management

Requirements change throughout the project lifecycle. Manage changes systematically:

  1. Change Request: Stakeholder submits formal request
  2. Impact Analysis: Assess cost, schedule, risk impacts
  3. Review: Change Control Board (CCB) evaluates
  4. Approval/Rejection: Decision with rationale
  5. Implementation: Update requirements, design, code, tests
  6. Verification: Ensure change implemented correctly

Version Control

Track requirement versions:

  • Document version numbers (1.0, 1.1, 2.0)
  • Change history (date, author, description)
  • Baseline requirements (approved snapshots)

Requirements Management Tools

  • Jira: User stories, epics, tracking
  • IBM DOORS: Enterprise requirements management
  • Azure DevOps: Integrated ALM platform
  • Confluence: Documentation and collaboration
  • Trello: Lightweight Kanban-based tracking

Practice Problems

Problem 1: Rewrite the following poor requirements to be clear, testable, and unambiguous:

a) "The website should load quickly"

b) "The system must be secure"

c) "Users should find the interface easy to use"

Solution:

a) "The website homepage shall load completely within 3 seconds on a standard broadband connection (10 Mbps) for 95% of requests"

b) "The system shall:

  • Encrypt all data in transit using TLS 1.3
  • Encrypt all data at rest using AES-256
  • Require passwords with minimum 12 characters including uppercase, lowercase, number, and symbol
  • Implement multi-factor authentication for admin accounts
  • Log all authentication attempts with timestamp and IP address"

c) "New users shall be able to complete their first product purchase within 5 minutes without consulting documentation, as verified by usability testing with 80% success rate"

Problem 2: Write a complete use case for "User registers for an account" including preconditions, main success scenario, and at least two extensions.

Use Case: Register User Account
Primary Actor: New User
Preconditions: User has valid email address and internet connection
Postconditions: User account created, confirmation email sent

Main Success Scenario:
1. User navigates to registration page
2. System displays registration form
3. User enters email, password, confirm password, first name, last name
4. User checks "I agree to Terms of Service" checkbox
5. User clicks "Register" button
6. System validates all fields
7. System checks email is not already registered
8. System hashes password using bcrypt
9. System creates user account in database
10. System sends confirmation email with activation link
11. System displays "Registration successful" message
12. Use case ends

Extensions:
6a. Validation fails:
    6a1. System highlights invalid fields in red
    6a2. System displays specific error messages:
        - "Email format is invalid"
        - "Password must be at least 12 characters"
        - "Passwords do not match"
    6a3. Return to step 3

7a. Email already registered:
    7a1. System displays "Email already exists. Try logging in or reset password"
    7a2. System provides links to login and password reset
    7a3. Use case ends

5a. Terms of Service not accepted:
    5a1. System displays "You must accept Terms of Service to register"
    5a2. Return to step 5

Problem 3: A system must be available 99.9% of the time. If the MTTR is 30 minutes, calculate the required MTBF.

Solution:

Using the availability formula:

$$\text{Availability} = \frac{\text{MTBF}}{\text{MTBF} + \text{MTTR}}$$

Given: Availability = 0.999, MTTR = 0.5 hours

$$0.999 = \frac{\text{MTBF}}{\text{MTBF} + 0.5}$$ $$0.999(\text{MTBF} + 0.5) = \text{MTBF}$$ $$0.999 \cdot \text{MTBF} + 0.4995 = \text{MTBF}$$ $$0.4995 = \text{MTBF} - 0.999 \cdot \text{MTBF}$$ $$0.4995 = 0.001 \cdot \text{MTBF}$$ $$\text{MTBF} = \frac{0.4995}{0.001} = 499.5 \text{ hours}$$

Answer: MTBF must be at least 499.5 hours (approximately 21 days)

Problem 4: Write 3 user stories with acceptance criteria for an e-commerce shopping cart feature.

Solution:

User Story 1:

As a shopper, I want to add items to my cart, so that I can purchase multiple products in one transaction

Acceptance Criteria:

  • Given I'm viewing a product, when I click "Add to Cart", then the item is added to my cart
  • Given I add an item, when the cart updates, then the cart icon shows the correct item count
  • Given I add the same item twice, when I view my cart, then the quantity is incremented (not duplicated)

User Story 2:

As a shopper, I want to adjust item quantities in my cart, so that I can change my mind about how many to buy

Acceptance Criteria:

  • Given I'm viewing my cart, when I change the quantity field, then the subtotal updates in real-time
  • Given I set quantity to 0 or click "Remove", when I confirm, then the item is removed from cart
  • Given the product has stock limits, when I exceed available quantity, then I see "Only X available" warning

User Story 3:

As a returning customer, I want my cart saved between sessions, so that I don't lose items if I leave the site

Acceptance Criteria:

  • Given I have items in my cart, when I log out and log back in, then my cart contents are preserved
  • Given I add items while logged out, when I log in, then guest cart merges with my saved cart
  • Given items in my cart are no longer available, when I view cart, then I see "Item no longer available" message

Key Takeaways

  • Requirements engineering is critical - poor requirements are the #1 cause of project failure
  • Functional requirements define WHAT the system does; non-functional requirements define HOW WELL it does it
  • Use cases provide detailed interaction scenarios; user stories are lightweight and Agile-friendly
  • Good requirements are complete, consistent, unambiguous, verifiable, and traceable
  • Elicitation requires multiple techniques (interviews, observation, prototyping) to gather complete requirements
  • Prioritization (MoSCoW, Kano, Value/Effort) ensures focus on high-impact features
  • Validation ensures requirements actually meet stakeholder needs before implementation
  • Requirements change - manage changes systematically with impact analysis and approval processes
  • Traceability matrices link requirements to design, code, and tests for complete coverage
