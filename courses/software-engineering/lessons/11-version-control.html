Version Control with Git

Master Git fundamentals, workflows, branching strategies, and best practices for collaborative software development.

1. Introduction to Version Control

Version control systems (VCS) track changes to files over time, enabling collaboration, history tracking, and the ability to revert to previous versions.

Why Version Control Matters:
  • History: Track all changes with who, what, when, why
  • Collaboration: Multiple developers work on same codebase
  • Branching: Develop features in isolation
  • Backup: Distributed copies protect against data loss
  • Experimentation: Try ideas without breaking main code
  • Code Review: Review changes before merging

Git: The most popular distributed version control system, created by Linus Torvalds in 2005.

3. Branching and Merging

3.1 Branch Basics

Branch Operations 
# List branches
git branch                     # Local branches
git branch -r                  # Remote branches
git branch -a                  # All branches

# Create branch
git branch feature-login       # Create but don't switch
git checkout -b feature-login  # Create and switch
git switch -c feature-login    # Create and switch (newer)

# Switch branches
git checkout main
git switch main                # Newer syntax

# Rename branch
git branch -m old-name new-name
git branch -m new-name         # Rename current branch

# Delete branch
git branch -d feature-login    # Safe delete (merged only)
git branch -D feature-login    # Force delete

# Delete remote branch
git push origin --delete feature-login

3.2 Merging Strategies

Merge vs Rebase 
# MERGE: Creates merge commit, preserves history
git checkout main
git merge feature-login

# Merge with no fast-forward (always create merge commit)
git merge --no-ff feature-login

# Abort merge if conflicts
git merge --abort

# REBASE: Rewrites history, creates linear timeline
git checkout feature-login
git rebase main

# Interactive rebase (edit, squash, reorder commits)
git rebase -i HEAD~3

# Continue after resolving conflicts
git rebase --continue

# Abort rebase
git rebase --abort

# When to use:
# MERGE: 
#   - Public branches
#   - Preserving complete history
#   - Team collaboration
#
# REBASE:
#   - Private branches
#   - Clean, linear history
#   - Before merging feature branch
Merge vs Rebase Visualization 
MERGE:
main:    A---B---C-------M
              \         /
feature:       D---E---F

git checkout main
git merge feature
# Creates merge commit M

REBASE:
main:    A---B---C
              \
feature:       D---E---F

git checkout feature
git rebase main

Result:
main:    A---B---C
                  \
feature:           D'---E'---F'
# Commits D, E, F are rewritten as D', E', F'

4.2 GitHub Flow

Simpler workflow focused on main branch and pull requests.

GitHub Flow Process 
1. Create a branch from main
   git checkout main
   git pull origin main
   git checkout -b feature/add-search

2. Make commits
   git add .
   git commit -m "Add search functionality"
   git push origin feature/add-search

3. Open Pull Request on GitHub
   - Describe changes
   - Request reviews
   - Run CI checks

4. Review and discuss
   - Address feedback
   - Make additional commits
   - Push to same branch

5. Deploy and test
   - Deploy to staging
   - Run tests

6. Merge to main
   git checkout main
   git merge feature/add-search
   git push origin main

7. Delete feature branch
   git branch -d feature/add-search
   git push origin --delete feature/add-search

5. Conflict Resolution

Resolving Merge Conflicts 
# Attempt merge
git merge feature-branch

# Conflict occurs
# CONFLICT (content): Merge conflict in file.txt
# Automatic merge failed; fix conflicts and then commit the result.

# Check conflicted files
git status

# File shows conflict markers:
<<<<<<< HEAD
Current branch content
=======
Incoming branch content
>>>>>>> feature-branch

# Resolve conflict manually:
# 1. Edit file to resolve differences
# 2. Remove conflict markers
# 3. Keep desired changes

# Mark as resolved
git add file.txt

# Complete merge
git commit -m "Merge feature-branch and resolve conflicts"

# Or abort if needed
git merge --abort

# Tools for conflict resolution
git mergetool              # Launch configured merge tool
git diff --conflicts       # Show conflicts
Conflict Resolution Strategies 
# Accept theirs (incoming changes)
git checkout --theirs file.txt
git add file.txt

# Accept ours (current branch)
git checkout --ours file.txt
git add file.txt

# View both versions
git show :1:file.txt       # Common ancestor
git show :2:file.txt       # HEAD (ours)
git show :3:file.txt       # MERGE_HEAD (theirs)

# Prevent conflicts:
# - Communicate with team
# - Pull frequently
# - Keep changes small
# - Merge main into feature branch regularly

6. Advanced Git Features

6.1 Stashing

Git Stash Commands 
# Save work in progress
git stash
git stash save "WIP: implementing user search"

# List stashes
git stash list
# stash@{0}: WIP: implementing user search
# stash@{1}: On main: emergency fix

# Apply stash
git stash apply            # Apply latest, keep in stash
git stash apply stash@{1}  # Apply specific stash

git stash pop              # Apply and remove from stash
git stash pop stash@{1}

# Create branch from stash
git stash branch feature-search stash@{0}

# Delete stash
git stash drop stash@{0}   # Delete specific
git stash clear            # Delete all

# View stash content
git stash show
git stash show -p          # With diff

6.2 Cherry-picking

Cherry-pick Commits 
# Pick specific commit from another branch
git cherry-pick abc123

# Pick multiple commits
git cherry-pick abc123 def456

# Pick range
git cherry-pick abc123..def456

# Pick without committing
git cherry-pick -n abc123

# Continue after conflict
git cherry-pick --continue

# Abort cherry-pick
git cherry-pick --abort

6.3 Rewriting History

Interactive Rebase 
# Edit last 3 commits
git rebase -i HEAD~3

# Interactive menu:
pick abc123 Add user model
pick def456 Fix typo
pick ghi789 Add tests

# Available commands:
# pick   - use commit
# reword - edit commit message
# edit   - edit commit content
# squash - merge with previous commit
# fixup  - like squash but discard message
# drop   - remove commit

# Example: squash commits
pick abc123 Add user model
squash def456 Fix typo
squash ghi789 Add tests
# Results in single commit

# Reorder commits
pick ghi789 Add tests
pick abc123 Add user model
pick def456 Fix typo

# Split commit
edit abc123
# Git pauses, reset commit
git reset HEAD^
# Make separate commits
git add file1.js
git commit -m "Part 1"
git add file2.js
git commit -m "Part 2"
git rebase --continue

6.4 Undoing Changes

Undo Operations 
# Undo working directory changes
git restore file.txt           # Discard changes
git checkout -- file.txt       # Old syntax

# Undo staging
git restore --staged file.txt
git reset HEAD file.txt        # Old syntax

# Undo last commit (keep changes)
git reset --soft HEAD~1

# Undo last commit (discard changes)
git reset --hard HEAD~1

# Undo specific commit (create new commit)
git revert abc123

# Recover deleted commit
git reflog                     # Find commit hash
git cherry-pick abc123         # Restore it

# Reset to remote state
git fetch origin
git reset --hard origin/main

# Clean untracked files
git clean -n                   # Dry run
git clean -f                   # Delete files
git clean -fd                  # Delete files and directories

7. Best Practices

Commit Message Guidelines 
# Good commit message structure:
(): 






# Example:
feat(auth): add OAuth2 authentication

Implement OAuth2 authentication flow using Google provider.
Users can now sign in with their Google accounts.

Closes #123

# Types:
feat:     New feature
fix:      Bug fix
docs:     Documentation
style:    Formatting
refactor: Code restructuring
test:     Adding tests
chore:    Maintenance

# Rules:
1. Use imperative mood: "Add feature" not "Added feature"
2. Keep subject under 50 characters
3. Capitalize subject line
4. No period at end of subject
5. Separate subject from body with blank line
6. Wrap body at 72 characters
7. Use body to explain what and why, not how
Git Best Practices 
# .gitignore - Ignore files
node_modules/
dist/
.env
*.log
.DS_Store

# Commit early, commit often
# Make small, logical commits

# Pull before push
git pull origin main
git push origin main

# Use branches for features
# Never commit directly to main

# Review before committing
git diff
git status

# Use tags for releases
git tag -a v1.0.0 -m "Release 1.0.0"
git push origin v1.0.0

# Sign commits (optional but recommended)
git config --global commit.gpgsign true
git commit -S -m "Signed commit"

8. Pull Requests and Code Review

Pull Request Workflow 
# 1. Create feature branch
git checkout -b feature/user-notifications
git push -u origin feature/user-notifications

# 2. Make changes and commit
git add .
git commit -m "Add email notification system"
git push origin feature/user-notifications

# 3. Create Pull Request on GitHub/GitLab
# - Fill in description
# - Link related issues
# - Request reviewers
# - Add labels

# 4. Address review feedback
# Make requested changes
git add .
git commit -m "Address review feedback"
git push origin feature/user-notifications

# 5. Squash commits before merge (optional)
git rebase -i main
# Squash all commits into one

# 6. Merge (on GitHub/GitLab or locally)
git checkout main
git pull origin main
git merge --no-ff feature/user-notifications
git push origin main

# 7. Delete branch
git branch -d feature/user-notifications
git push origin --delete feature/user-notifications
Code Review Checklist:
  • Code works and meets requirements
  • Tests included and passing
  • Follows coding standards
  • No security vulnerabilities
  • Performance considerations addressed
  • Documentation updated
  • Commit messages clear and descriptive

9. Common Pitfalls and Solutions

Common Git Problems 
# Problem 1: Committed to wrong branch
git log                        # Find commit hash
git checkout correct-branch
git cherry-pick abc123
git checkout wrong-branch
git reset --hard HEAD~1

# Problem 2: Need to undo pushed commit
git revert abc123              # Safe - creates new commit
git push origin main
# DON'T use reset on public branches!

# Problem 3: Forgot to pull before committing
git stash
git pull origin main
git stash pop
# Resolve conflicts if any

# Problem 4: Accidentally deleted branch
git reflog                     # Find commit
git checkout -b recovered-branch abc123

# Problem 5: Large file committed
git rm --cached large-file.zip
echo "large-file.zip" >> .gitignore
git commit -m "Remove large file"

# For history:
git filter-branch --tree-filter 'rm -f large-file.zip' HEAD

# Problem 6: Merge conflict seems impossible
git merge --abort
git pull origin main
# Try rebase instead
git rebase main

Key Takeaways

  • Git Basics: Master commits, branches, merging, and conflict resolution
  • Workflows: Choose appropriate workflow - GitFlow for releases, GitHub Flow for simplicity, Trunk-Based for CI/CD
  • Branching: Use feature branches, keep them short-lived, delete after merging
  • Merge vs Rebase: Merge for public branches, rebase for clean history on private branches
  • Commits: Small, logical commits with clear messages in imperative mood
  • Pull Requests: Essential for code review and collaboration
  • Never rewrite public history - use revert instead of reset on shared branches
  • Pull before you push, commit early and often
  • Use .gitignore to exclude build artifacts and sensitive files
