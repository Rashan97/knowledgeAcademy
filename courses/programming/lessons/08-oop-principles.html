Introduction to OOP

Object-Oriented Programming (OOP) is a paradigm based on the concept of "objects" that contain data (attributes) and code (methods). OOP promotes code reusability, modularity, and maintainability through four core principles: Encapsulation, Inheritance, Polymorphism, and Abstraction.

üìå The Four Pillars of OOP

  • Encapsulation: Bundling data and methods, hiding internal details
  • Inheritance: Creating new classes from existing ones
  • Polymorphism: Same interface, different implementations
  • Abstraction: Hiding complexity, showing only essential features

Classes and Objects

Python: Class Definition and Instantiation

class Dog: # Class attribute (shared by all instances) species = "Canis familiaris" def __init__(self, name, age): # Instance attributes (unique to each instance) self.name = name self.age = age def bark(self): return f"{self.name} says Woof!" def description(self): return f"{self.name} is {self.age} years old" # Creating objects (instances) buddy = Dog("Buddy", 3) max_dog = Dog("Max", 5) print(buddy.bark()) # Buddy says Woof! print(buddy.description()) # Buddy is 3 years old print(Dog.species) # Canis familiaris

JavaScript: Class Definition

class Dog { // Static property (shared by all instances) static species = 'Canis familiaris'; constructor(name, age) { // Instance properties this.name = name; this.age = age; } bark() { return `${this.name} says Woof!`; } description() { return `${this.name} is ${this.age} years old`; } } // Creating objects const buddy = new Dog('Buddy', 3); const maxDog = new Dog('Max', 5); console.log(buddy.bark()); // Buddy says Woof! console.log(Dog.species); // Canis familiaris

C++: Class Definition

#include <iostream> #include <string> using namespace std; class Dog { private: string name; int age; public: // Static member (shared by all instances) static const string species; // Constructor Dog(string n, int a) : name(n), age(a) {} // Methods string bark() { return name + " says Woof!"; } string description() { return name + " is " + to_string(age) + " years old"; } }; // Define static member const string Dog::species = "Canis familiaris"; int main() { Dog buddy("Buddy", 3); Dog maxDog("Max", 5); cout << buddy.bark() << endl; cout << Dog::species << endl; return 0; }

Encapsulation

Encapsulation bundles data and methods together while hiding internal implementation details. Access is controlled through public, private, and protected modifiers.

Python: Encapsulation with Name Mangling

class BankAccount: def __init__(self, account_number, balance): self.account_number = account_number # Public self._balance = balance # Protected (convention: single underscore) self.__pin = 1234 # Private (name mangling: double underscore) # Getter def get_balance(self): return self._balance # Setter with validation def deposit(self, amount): if amount > 0: self._balance += amount return True return False def withdraw(self, amount, pin): if pin == self.__pin and amount <= self._balance: self._balance -= amount return True return False # Property decorator (Pythonic getter/setter) @property def balance(self): return self._balance account = BankAccount("12345", 1000) print(account.balance) # 1000 (using property) account.deposit(500) print(account.get_balance()) # 1500 # print(account.__pin) # AttributeError - private!

C++: Access Modifiers

class BankAccount { private: string accountNumber; double balance; int pin; protected: void logTransaction(string type, double amount) { cout << type << ": $" << amount << endl; } public: BankAccount(string acc, double bal, int p) : accountNumber(acc), balance(bal), pin(p) {} // Getter double getBalance() const { return balance; } // Setter with validation bool deposit(double amount) { if (amount > 0) { balance += amount; logTransaction("Deposit", amount); return true; } return false; } bool withdraw(double amount, int userPin) { if (userPin == pin && amount <= balance) { balance -= amount; logTransaction("Withdrawal", amount); return true; } return false; } };

Inheritance

Inheritance allows creating new classes based on existing classes, promoting code reuse.

Python: Single Inheritance

class Animal: def __init__(self, name, age): self.name = name self.age = age def speak(self): pass # Abstract method def info(self): return f"{self.name} is {self.age} years old" # Dog inherits from Animal class Dog(Animal): def __init__(self, name, age, breed): super().__init__(name, age) # Call parent constructor self.breed = breed def speak(self): return f"{self.name} says Woof!" class Cat(Animal): def speak(self): return f"{self.name} says Meow!" dog = Dog("Buddy", 3, "Golden Retriever") cat = Cat("Whiskers", 2) print(dog.speak()) # Buddy says Woof! print(dog.info()) # Buddy is 3 years old (inherited) print(cat.speak()) # Whiskers says Meow!

Python: Multiple Inheritance

class Flyable: def fly(self): return "I can fly!" class Swimmable: def swim(self): return "I can swim!" # Multiple inheritance class Duck(Animal, Flyable, Swimmable): def speak(self): return f"{self.name} says Quack!" donald = Duck("Donald", 5) print(donald.speak()) # Donald says Quack! print(donald.fly()) # I can fly! print(donald.swim()) # I can swim!

C++: Inheritance and Access Specifiers

// Base class class Animal { protected: string name; int age; public: Animal(string n, int a) : name(n), age(a) {} virtual string speak() { return "Some sound"; } string info() { return name + " is " + to_string(age) + " years old"; } virtual ~Animal() {} // Virtual destructor }; // Derived class class Dog : public Animal { private: string breed; public: Dog(string n, int a, string b) : Animal(n, a), breed(b) {} // Override parent method string speak() override { return name + " says Woof!"; } }; class Cat : public Animal { public: Cat(string n, int a) : Animal(n, a) {} string speak() override { return name + " says Meow!"; } };

‚ö†Ô∏è Inheritance Best Practices

  • Use inheritance for "is-a" relationships (Dog IS-A Animal)
  • Prefer composition over inheritance when possible
  • Keep inheritance hierarchies shallow (2-3 levels max)
  • In C++, always make destructors virtual in base classes
  • Use the override keyword in C++ to catch errors

Polymorphism

Polymorphism allows objects of different types to be treated uniformly through a common interface.

Method Overriding (Runtime Polymorphism)

class Shape: def area(self): pass def perimeter(self): pass class Rectangle(Shape): def __init__(self, width, height): self.width = width self.height = height def area(self): return self.width * self.height def perimeter(self): return 2 * (self.width + self.height) class Circle(Shape): def __init__(self, radius): self.radius = radius def area(self): return 3.14159 * self.radius ** 2 def perimeter(self): return 2 * 3.14159 * self.radius # Polymorphism in action shapes = [Rectangle(5, 3), Circle(4), Rectangle(2, 7)] for shape in shapes: print(f"Area: {shape.area():.2f}") # Calls appropriate method

C++: Virtual Functions (Dynamic Polymorphism)

class Shape { public: virtual double area() = 0; // Pure virtual (abstract) virtual double perimeter() = 0; virtual ~Shape() {} }; class Rectangle : public Shape { private: double width, height; public: Rectangle(double w, double h) : width(w), height(h) {} double area() override { return width * height; } double perimeter() override { return 2 * (width + height); } }; class Circle : public Shape { private: double radius; public: Circle(double r) : radius(r) {} double area() override { return 3.14159 * radius * radius; } double perimeter() override { return 2 * 3.14159 * radius; } }; int main() { // Polymorphism with pointers Shape* shapes[] = {new Rectangle(5, 3), new Circle(4)}; for (int i = 0; i < 2; ++i) { cout << "Area: " << shapes[i]->area() << endl; delete shapes[i]; } return 0; }

Operator Overloading (C++)

class Complex { private: double real, imag; public: Complex(double r = 0, double i = 0) : real(r), imag(i) {} // Operator overloading Complex operator+(const Complex& other) { return Complex(real + other.real, imag + other.imag); } Complex operator*(const Complex& other) { return Complex(real * other.real - imag * other.imag, real * other.imag + imag * other.real); } bool operator==(const Complex& other) { return real == other.real && imag == other.imag; } friend ostream& operator<<(ostream& os, const Complex& c) { os << c.real << " + " << c.imag << "i"; return os; } }; int main() { Complex c1(3, 4), c2(1, 2); Complex c3 = c1 + c2; // Calls operator+ cout << c3 << endl; // 4 + 6i return 0; }

Abstraction

Abstraction hides implementation details and exposes only essential features.

Python: Abstract Base Classes

from abc import ABC, abstractmethod class PaymentProcessor(ABC): @abstractmethod def process_payment(self, amount): pass @abstractmethod def refund(self, transaction_id): pass class CreditCardProcessor(PaymentProcessor): def process_payment(self, amount): return f"Processing credit card payment: ${amount}" def refund(self, transaction_id): return f"Refunding transaction {transaction_id}" class PayPalProcessor(PaymentProcessor): def process_payment(self, amount): return f"Processing PayPal payment: ${amount}" def refund(self, transaction_id): return f"Refunding PayPal transaction {transaction_id}" # Cannot instantiate abstract class # processor = PaymentProcessor() # TypeError! processor = CreditCardProcessor() print(processor.process_payment(100))

JavaScript: Interfaces with Duck Typing

// JavaScript doesn't have formal interfaces, but uses duck typing class PaymentProcessor { processPayment(amount) { throw new Error('Must implement processPayment()'); } refund(transactionId) { throw new Error('Must implement refund()'); } } class CreditCardProcessor extends PaymentProcessor { processPayment(amount) { return `Processing credit card payment: $${amount}`; } refund(transactionId) { return `Refunding transaction ${transactionId}`; } } const processor = new CreditCardProcessor(); console.log(processor.processPayment(100));

Composition vs Inheritance

"Favor composition over inheritance" - design principle promoting flexibility.

# Composition: "has-a" relationship class Engine: def start(self): return "Engine started" def stop(self): return "Engine stopped" class Car: def __init__(self, model): self.model = model self.engine = Engine() # Composition: Car HAS-A Engine def start(self): return f"{self.model}: {self.engine.start()}" def stop(self): return f"{self.model}: {self.engine.stop()}" # Inheritance: "is-a" relationship class ElectricCar(Car): # ElectricCar IS-A Car def __init__(self, model, battery_capacity): super().__init__(model) self.battery_capacity = battery_capacity def charge(self): return f"Charging {self.battery_capacity}kWh battery" tesla = ElectricCar("Tesla Model 3", 75) print(tesla.start()) # Tesla Model 3: Engine started print(tesla.charge()) # Charging 75kWh battery

Practice Problems

Problem 1: Library Management System

Create a class hierarchy: Book (base), FictionBook, NonFictionBook. Implement methods for borrowing and returning with proper encapsulation.

class Book: def __init__(self, title, author, isbn): self.title = title self.author = author self._isbn = isbn self._is_borrowed = False def borrow(self): if not self._is_borrowed: self._is_borrowed = True return True return False def return_book(self): self._is_borrowed = False def info(self): return f"{self.title} by {self.author}" class FictionBook(Book): def __init__(self, title, author, isbn, genre): super().__init__(title, author, isbn) self.genre = genre class NonFictionBook(Book): def __init__(self, title, author, isbn, subject): super().__init__(title, author, isbn) self.subject = subject

Problem 2: Employee Management System

Create Employee (base), Manager, Developer classes. Implement polymorphic calculateSalary() method with different bonus structures.

class Employee { protected: string name; double baseSalary; public: Employee(string n, double s) : name(n), baseSalary(s) {} virtual double calculateSalary() = 0; virtual ~Employee() {} }; class Manager : public Employee { public: Manager(string n, double s) : Employee(n, s) {} double calculateSalary() override { return baseSalary * 1.5; // 50% bonus } }; class Developer : public Employee { public: Developer(string n, double s) : Employee(n, s) {} double calculateSalary() override { return baseSalary * 1.2; // 20% bonus } };

Problem 3: Shape Calculator with Composition

Create a DrawingBoard class that uses composition to contain multiple Shape objects and calculate total area.

class Shape { area() { throw new Error('Must implement area()'); } } class Circle extends Shape { constructor(radius) { super(); this.radius = radius; } area() { return Math.PI * this.radius ** 2; } } class Rectangle extends Shape { constructor(width, height) { super(); this.width = width; this.height = height; } area() { return this.width * this.height; } } class DrawingBoard { constructor() { this.shapes = []; } addShape(shape) { this.shapes.push(shape); } totalArea() { return this.shapes.reduce((sum, s) => sum + s.area(), 0); } }

Key Takeaways

  • Encapsulation bundles data and methods while controlling access through modifiers
  • Inheritance creates "is-a" relationships promoting code reuse
  • Polymorphism allows uniform treatment of different types through common interfaces
  • Abstraction hides implementation details exposing only essential features
  • Composition ("has-a") is often more flexible than inheritance ("is-a")
  • Virtual functions in C++ enable runtime polymorphism
  • Abstract classes define interfaces that subclasses must implement
  • Operator overloading extends polymorphism to operators in C++
